Agent 自进化框架

agent自进化框架

该项目为框架代码，因此只有简略且清晰的代码流程，之后可以根据不同应用场景进行填充和丰富，可扩展性搞。

整体流程设计

流程分为两个部分短期进化和长期进化。需要学生模型与教师模型，学生模型负责与用户的真实交流，教师模型负责确定进化方向。

短期进化

短期进化是对系统提示词（或者一些根本性的设定，看以后的场景应用，目前框架代码不应该涉及）的优化，不涉及模型参数，可解释性强。

长期进化

长期进化是根据模型的交互过程，构造新的训练数据集，然后进行RL训练，实现模型的参数化更新

##具体过程
学生模型：与用户对话，每次会话结束（后端监控30min没有新的交互，视为结束会话），将该轮的会话数据送给教师模型进行解析，同时存入本地记忆。之后的对话时，通过agentic rag进行检索历史相关记录（也就是构建一个检索工具，搭载本地mem0实现记忆或数据库）

教师模型：解析学生模型传来的会话数据。1. 将会话数据与学生模型的系统提示词进行对比，判断是否需要修改学生模型的系统提示词（此处应该有个通用的判断标准）；2. 将会话数据转化为 verl grpo 适合的RL数据格式（jsonl数据），可以得到多个json数据对象（即jsonl里的一行），可以作为以后RL训练集合。

快进化：学生模型的系统提示词修改；
慢进化：学生模型的模型权重参数的修改。当教师模型将某个会话数据全部转化为RL数据时，如果此时的RL数据条数大于1000条，则触发训练。比如，现在已经有999条RL数据，小于1000条，不进行RL 训练，教师模型解析完最后一个会话数据时，新增了100条数据，此时RL数据有1099条数据，大于1000条，进行训练。之后接入verl训练脚本，指向当前学生模型与新数据集，执行RL训练，得到新的模型权重，进行权重合并，得到新的学生模型，将其应用于用户（指针指向新模型路径？）

总结

快进化只修改系统提示词，慢进化只修改模型权重，其余不做改变，最终得到一个进化后的agent。该框架应该具备很强的可扩展性，方便后续使用时，可以快速应用于具体业务场景。

deepseek

重构后的框架设计方案

框架整体设计理念

本框架以**“快慢分离、教学相长、验证驱动”**为核心理念，将Agent的自进化分为两条异步并行的路径：

  • 快进化（Harness层进化） ：不修改模型权重，通过优化Agent的“行为规范集合”——包括提示词、工具策略、决策规则、记忆检索策略——来实现快速迭代。可解释性强，可随时回滚。

  • 慢进化（参数层进化） ：通过构造高质量RL数据集并进行GRPO训练，实现模型权重的参数化更新，从根本上提升模型能力。

系统部署两个核心角色：学生模型直接面向用户执行任务；教师模型在会话结束后进行复盘，负责生成进化提案和构造训练数据。

整个框架以接口化设计为原则，各模块之间仅通过标准化接口交互，确保后续能够灵活适配不同的业务场景和基础模型。

模块一：记忆与状态管理层

记忆系统采用三层架构：

  1. 短期记忆：存储当前会话的完整上下文，会话结束后归档。

  2. 中期记忆：基于向量数据库实现跨会话检索，支持Agent在对话中检索相关历史经验。检索策略本身可根据检索结果的相关性反馈进行动态调整。

  3. 长期记忆：将高频出现的成功模式提炼为结构化的“知识碎片”，以文本描述和可复用代码片段两种形式固化存储。记忆的提炼策略根据下游任务的成功率反馈持续优化。(依赖大模型做文本处理，本质是“经验编译”——把零散的、成功的交互轨迹（Trajectories） ，编译成结构化的可复用技能（Skills）。为Agent编写一部实时更新的“操作手册”，而不仅仅是“回忆录”。)

所有记忆操作均通过统一接口暴露，底层实现可替换（如从mem0切换为其他向量数据库）。

模块二：快进化——Harness层敏捷迭代

快进化的目标是优化Agent的“行为规范”，不触碰模型权重。

进化对象：Harness集合，包含：

  • 系统提示词与角色设定
  • 工具调用策略（调用时机、调用顺序、结果处理）
  • 决策规则（条件判断、优先级、兜底逻辑）
  • 记忆检索策略（检索时机、Top-K、重排序）

进化流程（每次会话结束后触发）：

  1. 经验蒸馏：教师模型解析会话数据，提取成功模式和失败模式。

  2. 提案生成：教师模型基于蒸馏结果，生成3-5个Harness修改候选方案。

  3. 提案验证：在近期历史会话的子集上回放各候选方案，计算预期收益（如任务成功率提升、延迟降低等）。

  4. 提案提交：只有验证通过（收益超过设定阈值）的方案才被采纳，记录新版本号、修改内容和验证结果。未通过的方案记录失败模式，供后续参考。

  5. 版本管理：每次提交生成新的Harness版本号，支持按需回滚到任一历史版本。

模块三：慢进化——参数层深度重塑

慢进化涉及模型权重的更新，目标是从根本上提升学生模型的能力上限。

数据构造（每次会话结束后）：

  1. 奖励信号生成：教师模型为会话生成多维奖励信号——

    • 结果奖励：任务是否成功完成
    • 过程奖励：推理链条中每一步的质量
    • 一致性奖励：行为是否符合预期模式

  2. 数据格式化：将对话轨迹与奖励信号融合，转化为GRPO兼容的JSONL格式。

  3. 质量过滤：每条数据需通过证据验证（可追溯的推理依据），低质量或重复数据自动丢弃。验证通过的数据进入训练缓冲区。

训练触发（复合条件）：

  • 高质量数据 ≥ 500条，或总数据 ≥ 2000条
  • 且距离上次训练 ≥ 设定的安全间隔
  • 且验证集性能未出现下降趋势

训练与部署：

  1. 调用verl训练脚本，指向当前学生模型与训练缓冲区
  2. 训练完成后进行权重合并
  3. 在验证集上执行A/B测试：新旧模型对比
  4. 性能提升 → 部署新模型，清空已训练数据
  5. 性能下降 → 自动回滚，记录失败原因

模块四：进化治理与安全保障

为确保自进化过程可控，框架内置以下治理机制：

  1. 可审计的进化 lineage：每次进化操作（快或慢）均记录：时间、触发原因、修改内容、验证结果、决策依据。完整可追溯。

  2. 自动回滚：快进化验证未通过或慢进化A/B测试失败时，自动回滚到上一稳定版本。

  3. 防遗忘保护：慢进化训练时混合旧数据重放，或在RL目标中加入约束项，防止新训练覆盖已有能力。

  4. 渐进式部署：新版本先在小流量用户群中试运行，确认无问题后再全量部署。

框架扩展性设计

为保证框架能快速应用于不同业务场景，采用以下设计原则：

  1. 接口化模块：学生模型、教师模型、记忆系统、验证器均通过抽象接口定义，具体实现可插拔替换。

  2. Harness可配置：Harness的具体组成（包含哪些维度的配置项）可通过配置文件定义，不同业务可定义不同的Harness结构。

  3. 奖励函数可定制：奖励信号的维度和权重可根据业务目标调整。

  4. 触发条件可配置：快进化的触发频率、慢进化的复合触发阈值均可通过配置调整。

  5. 教师模型可替换：教师模型可使用更强的LLM（如GPT-4、DeepSeek-V3），也可使用轻量级模型以降低成本，仅需实现统一的复盘接口。

框架流程总览

deepseek（简化版）

Agent自进化框架

一、整体架构

  • 学生模型：面向用户，装载当前的系统提示词和模型权重。
  • 教师模型：离线运行，在会话结束后触发，职责为：
    • 修改系统提示词（快进化）
    • 生成带奖励标注的训练数据（慢进化素材）
  • 记忆系统：两层——内存记忆（临时会话上下文）+ 向量库记忆（经验摘要检索）+记忆持久化（全量记忆）

二、数据集动态划分机制

所有会话数据按时间顺序持续累积，教师模型在每次触发进化时，按以下规则动态划分数据集：

数据分区定义（以当前累积总量为 T 条为基准）：

分区 数据范围 用途 大小
最新数据（训练集） 最近新增的连续会话 慢进化训练素材 N 条（可配置，默认1000，按会话边界截断，实际 ≥ N）
次新数据（验证集组成部分） 训练集之前的连续会话 验证集（评估泛化能力） M 条（可配置，默认1500）
历史数据池（验证集组成部分） 次新数据之前的所有数据 验证集（从全量历史中随机抽样，评估遗忘程度） K 条（可配置，默认500，从历史数据池中随机抽取）
中间数据（暂不使用） 历史数据池中未被抽到的部分 暂不参与任何进化 剩余部分

数据流转示例（假设当前已累积 6000 条）：

关键说明：

  • 这些数据集都会完整持久化下来，供以后使用。
  • 训练集是滚动窗口——训练完成后即清空，下一轮从新的“最新数据”重新累积。
  • 验证集由两部分组成：
    • 次新 M 条：紧邻训练集之前的数据，用于评估模型在新近分布上的泛化能力。
    • 历史随机 K 条：从次新数据之前的所有历史数据中随机抽样，用于监控模型对早期知识的遗忘程度。
  • 每次触发验证时，历史部分重新随机抽样，引入多样性，防止验证集固化。
  • 三个参数 N、M、K 均可配置，业务可根据场景灵活调节。

三、快进化——系统提示词优化

触发：每次会话结束后（或累计若干条后批量触发）。

流程：

  1. 诊断与生成：教师分析会话，生成一个修改后的新版系统提示词。
  2. 验证：在动态验证集（次新 M 条 + 历史随机 K 条）上对比新旧提示词的表现。
  3. 决策：
    • 新提示词得分 ≥ 旧提示词 → 执行替换，记录版本号
    • 新提示词得分 < 旧提示词 → 丢弃修改，记录失败原因

四、慢进化——模型权重优化

触发：训练集（最新数据）累积达到 N 条（按会话边界截断，可略大于 N）时触发。

流程：

  1. 数据构造：教师对每轮会话生成综合奖励信号（结果奖励 + 过程奖励），拼接为 GRPO 兼容的 JSONL 格式。
  2. 质量过滤：过滤奖励值过低或信息不完整的数据。
  3. 训练：调用 verl 脚本，基于当前学生权重 + 训练集（最新 N 条）执行 GRPO 微调。
  4. 验证：在动态验证集（次新 M 条 + 历史随机 K 条）上对比新旧权重。
  5. 决策：
    • 新权重得分 ≥ 旧权重 → 合并部署，清空训练集
    • 新权重得分 < 旧权重 → 回滚旧权重，清空训练集

六、参数配置说明

参数 默认值 说明 调参建议
N（训练集大小） 1000 触发训练所需的最新数据条数 数据量大可增大N，数据稀疏可减小N
M（次新验证集大小） 1500 用于评估泛化能力的次新数据条数 一般设置为 N 的 1.5 倍左右

七、扩展性配置（接入业务只需3项）

  1. 系统提示词模板：标记可变占位符，框架自动识别并替换。
  2. 验证集评估函数：业务方定义评分逻辑（规则匹配 / LLM评分），框架负责调用比对。
  3. 奖励评分函数：业务方定义教师如何为会话打分，框架只接收数值。

deepseek（学习价值分估计）

Agent自进化框架（价值驱动版）

一、整体架构

  • 学生模型：面向用户，装载当前的系统提示词和模型权重。
  • 教师模型：离线运行，在会话结束后触发，职责扩展为：
    • 修改系统提示词（快进化）
    • 生成奖励信号 + 学习价值分（慢进化素材筛选）
  • 记忆系统：两层——内存记忆（会话上下文）+ 向量库记忆（经验摘要检索）

二、学习价值分（核心新增机制）

教师模型在每次会话复盘时，除了生成奖励信号，还需输出一个学习价值分（0-1之间），用于衡量该条数据对模型进化的有用程度。

学习价值分的评估维度：

维度 说明 高分特征
结果重要性 任务是否关键、影响面大小 涉及核心业务逻辑、影响用户体验的关键任务
难度系数 任务的复杂程度 多步推理、多工具调用、需要处理异常边界
新颖程度 与历史经验的差异度 首次出现的场景、新的问题模式
不确定性 模型对该任务的把握程度 模型置信度低、推理路径不清晰
可学习性 模型是否有改进空间 当前处理有缺陷但方向明确

教师模型综合以上维度，输出一个0-1之间的标量分值，作为后续数据筛选和采样的依据。

三、数据集动态划分与价值采样机制

所有会话数据按时间顺序持续累积，每条数据均附带教师模型标注的学习价值分。

触发进化时的数据划分：

分区 数据来源 采样方式 大小
最新数据池 最近新增的连续会话 价值优先级采样：从最新数据中，按价值分从高到低优先选取 N1 条（可配置，默认1000，按会话边界截断）
历史数据池 最新数据池之前的所有数据 价值加权随机采样：价值分越高的数据，被抽中的概率越大 N2 条（可配置，默认1000）
训练集 最新N1条 + 历史随机N2条 合并组成训练集 N1 + N2 条
次新验证集 训练集数据来源之前的连续会话 直接取最近的一段 M1 条（可配置，默认1500）
历史验证集 次新验证集之前的所有数据 价值分层随机抽样：先按价值分分层（高/中/低），每层随机抽取，确保验证集覆盖各价值层级 M2 条（可配置，默认500）
验证集 次新 M1 条 + 历史分层随机 M2 条 合并组成验证集 M1 + M2 条

价值采样详细说明：

  1. 最新数据价值优先级采样：最新数据池中的数据先按学习价值分从高到低排序，优先选取高分数据。若高分数据不足N1条，则顺延至中等分数据。

    • 目的：确保训练集优先包含“最近发生的高价值经验”，而非盲目全量纳入。

  2. 历史数据价值加权随机采样：历史数据池中的每条数据被抽中的概率与其价值分成正比。具体为：P(被选中) = 价值分 / Σ(所有历史数据价值分)。

    • 目的：从全量历史中，让“经典难题”和“高价值案例”有更高概率被反复学习，防止遗忘。

  3. 历史验证集价值分层随机抽样：先将历史数据按价值分分为三层（高：0.7-1.0，中：0.3-0.7，低：0-0.3），然后从每层中按比例（如 50%-30%-20%）随机抽取，确保验证集能代表各价值层级的分布，避免验证集偏向某一类数据。

    • 目的：验证集能够全面覆盖从简单到困难、从常规到边缘的各种情况，使“验证通过”的结果更具说服力。

四、快进化——系统提示词优化

触发：每次会话结束后（或累计若干条后批量触发）。

流程：

  1. 诊断与生成：教师分析会话，生成一个修改后的新版系统提示词。
  2. 验证：在价值分层验证集（次新 M1 条 + 历史分层随机 M2 条）上对比新旧提示词的表现。评估指标为验证集上的平均得分。
  3. 决策：
    • 新提示词得分 ≥ 旧提示词 → 执行替换，记录版本号
    • 新提示词得分 < 旧提示词 → 丢弃修改，记录失败原因

五、慢进化——模型权重优化

触发：训练集数据量达到 N1 + N2 条时触发。

流程：

  1. 数据构造：教师对每轮会话生成综合奖励信号（结果奖励 + 过程奖励），拼接为 GRPO 兼容的 JSONL 格式。
  2. 价值标注：教师同时输出每条数据的学习价值分，存入数据元信息。
  3. 价值采样构建训练集：
    • 从最新数据池中，按价值优先级采样 N1 条
    • 从历史数据池中，按价值加权随机采样 N2 条
    • 合并生成最终训练集
  4. 训练：调用 verl 脚本，基于当前学生权重 + 训练集执行 GRPO 微调。
  5. 验证：在价值分层验证集（次新 M1 条 + 历史分层随机 M2 条）上对比新旧权重。
  6. 决策：
    • 新权重得分 ≥ 旧权重 → 合并部署，清空训练集
    • 新权重得分 < 旧权重 → 回滚旧权重，清空训练集

六、完整闭环工作流

七、参数配置说明

参数 默认值 说明
N1 1000 训练集中来自最新数据的条数（价值优先级采样）
N2 1000 训练集中来自历史数据的条数（价值加权随机采样）
M1 1500 验证集中次新数据的条数
M2 500 验证集中历史分层随机抽样的条数
价值分层比例 高:中:低 = 50%:30%:20% 历史验证集分层抽样时各层占比

八、扩展性配置（接入业务只需4项）

  1. 系统提示词模板：标记可变占位符，框架自动识别并替换。
  2. 验证集评估函数：业务方定义评分逻辑（规则匹配 / LLM评分）。
  3. 奖励评分函数：业务方定义教师如何为会话打分。
  4. 学习价值分函数：业务方定义教师如何评估数据的“学习价值”（可基于难度、新颖度、不确定性等维度组合）。这是该框架新增的扩展点。

九、与原简化版的对比

维度 原简化版 价值驱动版
训练集采样 最新N1条 + 历史随机N2条 最新N1条（价值优先级）+ 历史N2条（价值加权随机）
验证集历史部分 最旧数据随机M2条 全量历史分层随机M2条（覆盖高/中/低价值）
数据质量保障 无 学习价值分综合评估5个维度
遗忘监控 仅监控最旧数据 监控全量历史的分层抽样
扩展点 3个 4个（新增价值分函数）

十、核心优势总结

设计点 效果
价值优先级采样（最新数据） 确保高价值的新经验优先被学习
价值加权随机采样（历史数据） 让经典难题有更高概率被反复巩固
价值分层验证集 验证集覆盖各难度层级，验证结果更可信
学习价值分可定制 业务方可定义“什么是有价值的经验”
训练集用完即弃 防止旧数据干扰策略偏移

这个版本在保持原有简洁性的同时，将数据采样从“基于时间和数量”升级为“基于价值和数量”，实现了从“随机填鸭”到“精准进补”的进化。可以开始写核心调度代码了吗？
