4B 小模型 Agentic RL 实验

实验概述

本次实验聚焦小参数量模型的工具调用能力优化，以专家类计算任务为示例，探索 4B 规模模型通过“监督微调（SFT）冷启动 + 基于可验证奖励的强化学习（RLVR）“路径习得工具调用能力的效果与优化方向。实验基于 Qwen3-4B 基座，构建了匹配不同训练阶段的数据集与工具执行环境，完整复现了从冷启动到策略优化的全流程。

基础训练配置

  • 算力资源：4 张 A100 显卡
  • 基座模型：Qwen3-4B
  • SFT 框架：XTuner
  • RL 框架：VERL
  • 核心 RL 算法：GRPO（Group Relative Policy Optimization，分组相对策略优化）。通过同 prompt 下多 rollout 的组内相对奖励估计优势，无需单独训练 Critic 价值网络，是当前 RLVR 范式下的主流对齐算法。

多阶段实验设计

实验采用递进式三阶段设计，核心变量为 SFT 数据的工具标签占比与奖励函数的约束维度，逐步验证工具调用能力的习得规律与优化空间：

  • 第一阶段：生成无 <tool_call> 标签的 SFT 数据（10000 条）用于冷启动，作为纯知识求解的基线
  • 第二阶段：生成有 <tool_call> 标签的 SFT 数据（10000 条）用于冷启动，验证显式工具格式对能力的提升
  • 第三阶段：生成含/不含 <tool_call> 标签的 SFT 混合数据（测试 5:5、8:2 等比例）用于冷启动，验证混合策略+约束奖励的综合效果

SFT 数据构建逻辑

数据集构造流程

  1. 模板设计：构建 10 套不同的带工具提示词模板，对模板中的数值、参数位进行挖空处理
  2. 批量填充：通过预设工具函数自动补全模板空缺，生成多样化的训练样本，保证数据分布的丰富性
  3. 格式对齐：统一转换为对话式训练格式，适配 XTuner 微调框架

SFT 数据格式

{
  "conversation": [
    {
      "system": "你是一个有用的助手",
      "input": "用户的问题",
      "output": "模型的回答"
    }
  ]
}

RL 数据构建逻辑

数据集构造流程

  1. 提示词模板与 SFT 阶段保持一致，保证任务分布对齐，避免分布偏移
  2. 标准答案（ground_truth）通过工具执行自动生成，用于后续规则化奖励校验
  3. 共生成 10000 条 RL 训练数据，覆盖与 SFT 同分布的专家任务

RL 数据格式

适配 VERL 框架的强化学习训练格式，内置奖励模式与真值字段：

RL 多段奖励函数设计

本次实验采用 RLVR（Reinforcement Learning with Verifiable Rewards，基于可验证奖励的强化学习） 范式，所有奖励均通过规则化自动校验完成打分，无需人工标注或额外奖励模型，是当前推理类、工具调用类任务强化学习的主流方案。

奖励函数采用三段式递进设计，分别约束工具调用格式、答案正确性与调用效率，与业界 GRPO 训练的经典“格式+功能+效率“三层奖励设计思路一致。

第一部分：工具调用校验（可选）

针对带 <tool_call> 标签的训练阶段（第二、三阶段）启用。若模型输出未包含合法的 <tool_call> 标签，直接返回 0 分，强制模型学习工具调用的格式规范。

第二部分：任务正确性评分（必选）

针对具体任务的答案准确性进行打分，是奖励的核心主体。不同任务可根据自身校验逻辑适配不同的打分规则。

第三部分：冗余调用惩罚（可选）

仅当答案完全正确（前序得分为 1）时触发。逻辑为：对模型输出进行字符串匹配，定位第一次出现正确答案的位置，统计答案之后出现的冗余 <tool_call> 次数，采用指数衰减方式扣分——冗余调用越多，扣分越多，激励模型减少无效调用、提升执行效率。

核心观测指标

指标 含义与作用
num_turns/mean 平均对话轮次，用于判断模型是否执行工具调用、是否存在冗余调用。计算规则：num_turns = 用户轮次 + 助手轮次 + 1（初始用户提问）
critic/rewards/mean 批次平均奖励得分，反映模型整体策略的优劣程度
critic/advantages/mean 优势函数均值，用于判断收敛状态；当批次内奖励全 0 或全 1 时，优势值为 0，模型停止有效更新
acc/mean@1 验证集任务准确率，是衡量模型任务求解能力的核心效果指标

阶段分析

混合数据可以让模型同时习得“直接回答”与“工具调用”两种求解策略，但 5:5 比例下最终准确率略低于纯工具数据；当前指数惩罚的力度不足以抵消“多调用提升正确率”的收益，模型最终仍会收敛到冗余调用策略。

实验结论与优化方向

受时间限制，本次仅完成基础梯度实验，以下为基于现有结果的总结与后续优化建议。

数据层面

  1. 混合数据具备可行性：含/不含工具标签的混合数据可让模型习得两种求解策略，具备实用价值，但需进一步调试最优比例（可继续测试 6:4、7:3 等梯度）。
  2. 数据打乱是必要操作：若混合数据不打乱顺序，模型在前半段无标签数据上学会“不调用工具”后，后半段带标签数据训练时可能因工具校验奖励全 0，导致优势为 0、梯度消失，模型无法更新。
    • GRPO 算法：全 0/全 1 奖励会导致组内优势归一化后失效，梯度为 0，模型停止学习，这是 GRPO 算法的已知固有缺陷。
    • DAPO 算法：全称为 Decoupled Clip and Dynamic sAmpling Policy Optimization（解耦裁剪与动态采样策略优化），通过提升 clip ratio 上限、引入重试策略缓解 GRPO 的梯度消失与熵塌陷问题。但在模型已形成稳定策略的场景下，其探索能力仍有限，难以打破已固化的策略偏好。
    • OPD 算法：全称为 On-policy Distillation（在线策略蒸馏），通过外部教师模型提供监督信号，不受学生模型当前策略的限制，更有可能打破策略固化，理论上效果优于前两者。

训练层面

可采用早停策略平衡效果与效率。例如第三阶段实验中，训练至 800 步左右即可停止，此时准确率已达 90% 左右，且 num_turns 仍处于动态优化状态，可避免最终收敛到冗余调用策略，兼顾准确率与调用效率。

奖励设计层面

当前三段式奖励框架整体合理，但需精细化调参以平衡多个目标：

  • 调整工具调用校验的严格程度，避免过度约束抑制探索
  • 优化冗余调用的惩罚系数，平衡“答案正确性”与“调用效率”的权重
  • 可增加对最少调用步数的正向奖励，主动引导模型用最少轮次得到正确答案

奖励函数核心实现

@classmethod
def _verify_correction(cls, extract_solution, identity: dict, **kwargs) -> float:
    """
    验证提取的解决方案并计算正确性分数。
    现在 extract_solution 可能是一个字典（包含 parsed, raw_output, end_pos, parsed_str）。
    """
    # 提取答案相关信息
    if isinstance(extract_solution, dict):
        parsed = extract_solution["parsed"]
        raw_output = extract_solution.get("raw_output", "")
        end_pos = extract_solution.get("end_pos", len(raw_output))
        parsed_str = extract_solution.get("parsed_str", "")
    else:
        parsed = extract_solution
        raw_output = ""
        end_pos = 0
        parsed_str = ""
    if not parsed or not isinstance(parsed, list) or len(parsed) != 2:
        return 0.0
        
    try:
        # 第一部分：检查是否调用工具
        if not "<tool_call>{\"name\": \"detect_parallel_intervals\", \"arguments\": {\"current_data\": [{" in raw_output or not "</tool_call>" in raw_output:
            return 0.0
        # 第二部分：根据不同应用场景调整奖励设计参数
        final_score = max(0.0, 1.0 - penalty)
        # 第三部分：校验答案后多余tool call
        # 仅满分（完全正确）时，校验答案后多余tool call
        if final_score >= 1.0 and raw_output and parsed_str:
            # 找到第一个标准答案字符串的末尾下标
            ans_start_idx = raw_output.find(parsed_str)
            if ans_start_idx != -1:
                ans_end_idx = ans_start_idx + len(parsed_str)
                # 截取答案字符串之后全部文本
                text_after_answer = raw_output[ans_end_idx:]
                # 统计答案后所有工具调用标记
                tool_call_count = text_after_answer.count("<tool_call>")
                if tool_call_count > 0:
                    # 指数惩罚，每多一次tool call乘以0.95
                    final_score *= (0.95 ** tool_call_count)
        return max(0.0, final_score)
    except Exception as e:
        print(f"  [验证错误] {e}")
        import traceback
