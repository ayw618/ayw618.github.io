Agent模型评测

Agent框架

该Agent模型框架如下：

与用户交互的模型为意图识别模型（主模型），其它模型作为工具调用模型。大致流程如下：

  • 用户输入的指令送入意图识别模型。
  • 意图识别模型根据指令的意图调用对应的工具，部分工具需要模型来进行处理，例如代码生成工具。
  • 工具执行完毕，会返回工具执行结果给意图识别模型，比如，工具执行成功的日志、错误信息等。
  • 意图识别模型根据工具执行结果，可以再次调用工具。
  • 意图识别模型通过循环调用工具，直到完成指令的执行。
  • 最终，意图识别模型返回执行结果，并总结执行过程给用户。

    个人认为 Agent 核心主要有三部分：模型、工具、循环。
    模型作为大脑、工具作为手/脚、循环作为各种感官（反馈机制）。

当前的模型测评主要针对意图识别模型与代码生成模型。

意图识别模型测评

测评目的

选出一个意图识别模型，兼顾成本与效果。评测模型的意图识别与工具调用能力。

测评方法

将意图识别模型的测评放在真实的Agent链路内进行，除了将部分工具调用的执行过程进行简化，其它Agent环境配置不变，比如，系统提示词、工具列表信息等。
当前测评任务仅对代码生成工具的返回结果进行了简化，该工具执行过程不包含实际代码生成模型，只返回代码生成工具的虚假执行结果。
其余工具执行快，可以不做简化。

@tool
def code_generator(name: str, description: str, **kwargs) -> str:
    """代码生成工具：输入代码描述，返回代码执行结果"""
    # 此处为真实的代码生成模型调用
    # info = generate_code(name=name, description=description) # 执行意图识别模型评测时，注释该行
    info = "代码生成工具执行成功" # 在真实生产环境时，注释该行
    return f"代码生成工具执行结果：{info}"

测评流程

以下是意图识别模型的测评流程：

  • 基准模型：用于打分与评测，不考虑成本。
  • 待测模型：需要测评的意图识别模型。用于后续实际业务的应用落地，倾向于成本低的模型。

    由于实际业务有成本考虑，所以先利用小批次评测数据对各种强大模型（不考虑成本）进行快速筛选，选出效果较好的基准模型。
    然后，再利用基准模型对成本较低、开源的模型进行大批量数据评测，选出最终可以落地的意图识别模型。

测评指标

  • 意图识别：
    • 奖励：识别出一个意图得1分
    • 惩罚：未识别出一个意图得0分，每个意图识别错误得0分
  • 工具调用：
    • 奖励：成功调用一个必须的工具得1分（如果不需要工具，则无需调用，直接得满分）
    • 惩罚：未调用一个必须的工具得0分
  • 执行速度：记录运行的时间，单位为秒。用于参考

基准模型

  • 数据构建：通过人工构建20条复杂的指令，每个指令包含多个意图（2个及以上的意图）。
  • 评测方式：对多个强大模型（如GPT-5.5、Claude-opus-4.6等）进行人工评测（评测标准不变），选出意图识别最好的模型。
    复杂指令示例：
你好，你是谁？改一下code/readme.md文件。这个<专有名词>是什么意思呢？写个贪吃蛇游戏吧
  • 实验结果：
    测试过程中，发现部分模型在识别这一类复杂意图时，会出现一种现象：在Agent执行完毕，最终输出总结回答时，只包含指令中的最后一个意图；但是，当修改指令提示词（其余Agent系统提示词等均不变），要求模型进行意图识别时，模型都会正确识别出指令中的所有意图。
识别以下指令中的意图：
你好，你是谁？改一下code/readme.md文件。这个<专有名词>是什么意思呢？写个贪吃蛇游戏吧

这也证实了，当评测意图识别模型时，不能仅仅测试单条指令的意图识别的准确率，需要将其放入真实的Agent链路中进行测试。
真实的Agent链路是一个长链路，意图识别可能只是链路头部的一个节点，在后续的链路节点中，模型可能接收到大量工具调用的执行结果。到最后的总结回答节点时，部分模型在后续信息的影响下（噪声？），可能会忽略部分简单的意图。
比如，对于复杂指令示例，模型执行到最后总结回答时，只说明了贪吃蛇游戏的内容，前面的意图（比如，你是谁？改一下code/readme.md文件）都被忽略了。

最终选择的基准模型假定为模型 A。

代码生成模型评测

测评目的

选出一个代码生成模型，兼顾成本与效果。评测模型的在特定业务场景下的代码生成能力。

本人参与的项目中，生成的代码框架是固定的，因此，该评测就是为了选出一个适合该业务场景的代码生成模型。

测评方法

单独测试该工具code_generator的代码生成能力，不考虑其他工具的调用。

@tool
def code_generator(name: str, description: str, **kwargs) -> str:
    """代码生成工具：输入代码描述，返回代码执行结果"""
    # 此处为真实的代码生成模型调用
    info = generate_code(name=name, description=description) # 执行意图识别模型评测时，注释该行
    # info = "代码生成工具执行成功" # 在真实生产环境时，注释该行
    return f"代码生成工具执行结果：{info}"

测试流程

以下是代码生成模型的测评流程：

不再需要基准模型，因为该业务场景里有特定的脚本，可以直接测试代码文件是否符合业务要求。

测试指标

在该业务场景里，固定生成5个代码文件（根据不同任务，具体实现逻辑不同，但是代码框架一致），这些代码文件可以被一个固定框架的脚本串联起来。

评测指标为：

  • 语法正确率：模型生成的5个代码文件是否符合语法语法规范。每个文件 1 分，一共 5 分。
  • 脚本执行：脚本是否可以执行成功。该部分一共 5 分。
  • 代码生成时间：模型生成代码的时间，单位为秒。用于参考

评测步骤如下：

  1. 构建数据：这部分的数据由之前得到的意图识别模型B进行生成。
{
    "input": {
        "name": "不同任务名字",
        "description": "不同任务描述"
    }
}
  1. 调用不同的待测试模型，对相同的测试数据进行测试，获取测试结果。
  2. 列出评估结果表格
{
    "input": {
        "name": "不同任务名字",
        "description": "不同任务描述"
    },
    "output": {
        "file1": "代码文件1的路径",
        "file2": "代码文件2的路径",
        "file3": "代码文件3的路径",
        "file4": "代码文件4的路径",
        "file5": "代码文件5的路径"
    },
    "score": {
        "syntax": 5,
        "script": 5,
        "generate_time": 100
    }
}

综合得分 = (语法正确率 + 脚本执行)/10
由于脚本执行只有成功与失败两种状态，因此，综合得分只会出现 [0, 0.5]∪{1.0}
