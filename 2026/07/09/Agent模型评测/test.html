Agent模型评测

随着大语言模型 Agent 技术的产业落地，如何在效果、成本与业务适配性之间平衡选型，成为 Agent 系统落地的核心问题。不同于通用大模型的单轮能力评测，Agent 是”推理调度-工具执行-结果反馈”的闭环系统，其能力表现高度依赖真实交互链路。本文针对自研的工具调用型 Agent 框架，分别对核心调度层的意图识别模型与执行层的代码生成模型开展专项评测，探索兼顾效果与成本的落地选型方案。

一、Agent 整体架构

本 Agent 系统采用”集中调度 + 专用工具”的分层设计，整体框架如下：

其核心设计思路与经典 ReAct（Reasoning + Acting） 范式一致：通过”推理决策-工具执行-结果反馈”的多轮循环完成复杂任务。其中与用户直接交互的意图识别模型是整个系统的“大脑”，负责理解指令、拆解任务、调度工具；其余专用模型（如代码生成模型）被封装为标准化工具，负责具体任务的执行，这一设计也与 Toolformer 提出的”模型+工具”能力扩展思路一脉相承。

完整执行流程如下：

  1. 用户指令输入至意图识别模型，模型解析指令意图并选择对应工具调用
  2. 工具执行后将结果（成功日志、错误信息等）返回给意图识别模型
  3. 意图识别模型根据工具返回结果，判断是否需要继续调用工具
  4. 模型循环执行”调用工具-接收反馈”的流程，直到任务完成
  5. 最终由意图识别模型整合执行结果，向用户输出总结回答

个人认为 Agent 核心主要有三部分：模型、工具、循环。
模型作为大脑负责决策推理，工具作为手脚负责具体执行，循环作为反馈机制承接环境输入。这一划分与业界 Agent 系统的三大核心组件（推理模块、工具集、交互闭环）完全契合。

本次评测主要针对系统中的两个核心模型开展：作为调度核心的意图识别模型，以及作为核心执行工具的代码生成模型。

二、意图识别模型评测

2.1 评测目标

选出兼顾成本与效果的意图识别模型，核心考察模型在真实 Agent 链路中的多意图识别能力与工具调度准确性。作为整个 Agent 系统的调度核心，其能力直接决定了系统的任务完成率。

2.2 评测设计思路

当前很多 Agent 能力评测采用静态单轮指令的方式，但根据 AgentBench 等业界评测基准的结论：脱离真实交互链路的静态评测往往会高估模型的实际 Agent 能力——多轮工具返回的上下文会对模型的指令跟随能力产生干扰，也就是常说的”链路分布偏移”问题，静态测试无法反映这种真实场景下的能力衰减。

因此本次评测采用端到端真实链路评测方案：除代码生成工具为控制变量做了 Mock 简化外，其余 Agent 环境（系统提示词、工具列表、交互逻辑）完全与生产环境一致，确保评测结果能反映模型的真实上线表现。

其中代码生成工具的简化实现如下，仅返回固定成功结果，不触发真实代码生成，排除工具执行差异对意图识别评测的干扰：

@tool
def code_generator(name: str, description: str, **kwargs) -> str:
    """代码生成工具：输入代码描述，返回代码执行结果"""
    # 此处为真实的代码生成模型调用
    # info = generate_code(name=name, description=description) # 执行意图识别模型评测时，注释该行
    info = "代码生成工具执行成功" # 在真实生产环境时，注释该行
    return f"代码生成工具执行结果：{info}"

2.3 评测流程

考虑到强模型成本高、开源候选模型数量多的特点，本次采用”两阶段筛选”的评测范式，这也是当前工业界大规模模型选型的主流方案：

  1. 第一阶段：小样本人工评测，从顶级闭源模型中选出效果最优的作为基准裁判模型
  2. 第二阶段：以基准模型为评判标准，采用 LLM-as-Judge 范式，大批量自动化评测开源候选模型，选出性价比最优的落地模型

2.4 评测指标

本次评测从意图识别、工具调用两个核心维度进行量化打分，同时记录执行耗时作为成本参考：

  • 意图识别得分
    • 奖励：正确识别1个意图得1分
    • 惩罚：漏识别1个意图得0分，识别错误1个意图得0分
  • 工具调用得分
    • 奖励：成功调用1个必须工具得1分（无需工具的场景直接得满分）
    • 惩罚：漏调用1个必须工具得0分
  • 执行速度：记录单条指令的完整链路执行时间，单位为秒，作为成本参考

2.5 基准模型筛选

数据构建

人工构建20条复杂多意图指令，每条指令包含2个及以上不同类型的意图，覆盖闲聊、文件编辑、知识查询、代码生成等多种场景。
复杂指令示例：

你好，你是谁？改一下code/readme.md文件。这个<专有名词>是什么意思呢？写个贪吃蛇游戏吧

评测方式

对 GPT-5.5、Claude-opus-4.6 等多个顶级闭源模型进行人工评测，采用统一的打分标准，选出意图识别与工具调度能力最优的模型作为基准裁判。

关键实验发现

测试过程中观察到一个典型现象：部分模型在单轮”请识别以下指令的意图”的提问中，能准确识别所有意图；但放入完整 Agent 链路中，经过多轮工具返回后，最终总结回答只会覆盖最后一个意图，前面的简单意图会被完全忽略。

这一现象验证了长上下文交互中的指令跟随衰减效应：随着上下文长度增加、中间信息干扰增多，模型对早期用户指令的记忆与遵循能力会下降。这也进一步证实：评测意图识别模型不能只做单轮静态测试，必须放入完整 Agent 链路中进行端到端验证。

最终选定效果最优的模型 A 作为基准裁判模型。

2.6 开源待测模型评测

数据构建

利用多个强闭源模型生成100条符合格式的标注指令数据，每条数据包含用户输入与对应的意图、工具真值：

{
    "input": "你好，你是谁？改一下code/readme.md文件。这个<专有名词>是什么意思呢？写个贪吃蛇游戏吧",
    "ground_truth": {
        "intents": [
            "chat",
            "edit",
            "search",
            "code generate"
        ],
        "tools": [
            "edit",
            "search",
            "code_generate"
        ]
    }
}

评测步骤

对 Qwen3、Deepseek 等多个开源候选模型进行自动化评测，流程如下：

  1. 将 input 字段作为输入，在完整 Agent 链路中运行待测模型
  2. 提取模型最终的总结回答 output，同时硬编码采集执行过程中实际调用的工具列表 tools
  3. 将 output 输入基准模型 A，由基准模型识别其中包含的意图，得到 intents 结果
  4. 将识别出的 intents、tools 与 ground_truth 进行对比，计算准确率
  5. 输出完整评测结果的 jsonl 数据集

单条评测结果格式示例：

三、代码生成模型评测

3.1 评测目标

选出适配业务场景的代码生成模型，兼顾生成效果与调用成本。
本次业务场景中代码生成的框架是固定的，模型只需在既定框架内填充业务逻辑，因此评测重点不是通用算法能力，而是业务框架适配性与可执行性。

3.2 评测设计思路

通用代码生成评测（如 HumanEval、MBPP 等基准）侧重算法题的正确性，适合通用代码能力评估。但在垂直业务场景中，代码需要嵌入固定工程框架、通过既定脚本串联运行，因此采用定制化自动化评测方案：直接通过业务脚本校验代码的语法合规性与集成运行能力，无需人工评测，也无需额外基准模型，判分完全可复现，本质上属于”可验证奖励”的评测思路。

评测仅针对 code_generator 工具本身，隔离其他工具调用的干扰，专注测试代码生成能力。

@tool
def code_generator(name: str, description: str, **kwargs) -> str:
    """代码生成工具：输入代码描述，返回代码执行结果"""
    # 此处为真实的代码生成模型调用
    info = generate_code(name=name, description=description) # 执行代码生成模型评测时启用
    # info = "代码生成工具执行成功" # 在真实生产环境时，注释该行
    return f"代码生成工具执行结果：{info}"

3.3 评测流程

整体采用”统一输入-多模型生成-自动化校验”的流程，全程无需人工介入：

3.4 评测指标

该业务场景下，每个任务固定生成5个代码文件（不同任务实现逻辑不同，但代码框架完全一致），最终由统一的业务脚本串联执行。评测指标如下：

  • 语法正确率：5个代码文件是否符合语法规范，每个文件1分，满分5分
  • 脚本执行通过率：业务脚本是否能完整执行成功，满分5分
  • 代码生成时间：模型生成全部代码的耗时，单位为秒，作为成本参考

综合得分计算公式：

综合得分 = (语法正确率得分 + 脚本执行得分) / 10

由于脚本执行只有成功/失败两种状态，因此综合得分的取值范围为 [0, 0.5] ∪ {1.0}。

3.5 评测步骤与结果

  1. 测试数据构建：使用前述选出的意图识别模型 B 生成测试用例，包含任务名称与任务描述
{
    "input": {
        "name": "不同任务名字",
        "description": "不同任务描述"
    }
}
  1. 多模型并行测试：调用不同待测模型对同一批测试数据生成代码，记录生成结果与耗时
  2. 自动化校验打分：通过语法检查脚本与业务集成脚本自动校验，输出得分

单条评测结果格式示例：

{
    "input": {
        "name": "不同任务名字",
        "description": "不同任务描述"
    },
    "output": {
        "file1": "代码文件1的路径",
        "file2": "代码文件2的路径",
        "file3": "代码文件3的路径",
        "file4": "代码文件4的路径",
        "file5": "代码文件5的路径"
    },
    "score": {
        "syntax": 5,
        "script": 5,
        "generate_time": 100
    }
}

四、评测总结

本次针对 Agent 系统的两个核心模型完成了专项评测，整体形成了一套”端到端链路评测 + 两阶段 LLM-as-Judge 筛选 + 垂直场景定制化校验”的工业界 Agent 模型选型方法论，核心结论如下：

  1. Agent 模型评测必须放在真实交互链路中进行，单轮静态评测会高估模型能力，无法反映长上下文下的指令跟随衰减问题
  2. ”强模型做裁判 + 开源模型做候选”的两阶段评测方案，能很好地平衡评测质量与落地成本
  3. 垂直业务场景的代码生成评测，定制化的可执行校验比通用算法基准更有实际落地价值

最终选型结果：意图识别调度层采用开源模型 B，代码生成执行层采用模型 D，兼顾效果、成本与业务适配性。
